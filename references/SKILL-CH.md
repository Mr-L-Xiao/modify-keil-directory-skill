# Modify Keil Directory 技能说明（中文版）

本文档是 [SKILL.md](../SKILL.md) 的中文说明版，用于帮助维护者理解该 skill 的用途、触发条件、执行流程和资源组织方式。

## 技能用途

该 skill 用于规范化单工程 `Keil + MCU` 项目的本地目录结构和 `uVision` 工程分组结构，适用于：

- 裸机工程
- RTOS 工程
- 新建标准目录结构
- 修改已有 Keil 工程的目录结构
- 调整 Keil 工程分组
- 调整编译输出目录
- 复制可选的公共工具源码模板
- 生成根目录 `readme.md`
- 提供 Keil 输出路径和 `fromelf` 配置指导

## 适用场景

当用户提出以下需求时，应使用此 skill：

- “帮我规范一下这个 Keil 工程目录”
- “把这个 MCU 工程改成统一的本地目录结构”
- “顺便把 Keil 的 Group 也整理一下”
- “我要区分 RTOS 和无 RTOS 的目录模板”
- “帮我把输出文件放到 `Output/` 下面”
- “帮我给工程补一个目录结构说明文档”

## 执行流程

1. 在实际改动任何文件之前，必须先完成下文“必须确认的结构化选项”中的确认项。
2. 先确认目标是否为单工程 Keil 工程，而不是 `boot + app`、`master + slave` 之类的多工程结构。
3. 确认目标工程是否使用 RTOS。
4. 确认当前任务是：
   - 在空目录中创建标准结构，还是
   - 重构已有工程目录。
5. 确认输出方式：
   - 原地修改并备份，还是
   - 复制到新目录后再生成新结构。
6. 如果当前运行环境支持结构化交互工具和可点击选项框，则优先使用它们向用户展示选择项，而不是要求用户手动打字输入。
7. 如果当前环境不支持结构化选择框，则退化为简短的文本选择题。
8. 在实际修改前，先阅读 [layout-spec.md](layout-spec.md)。
9. 按规范应用本地目录结构。
10. 如果用户要求同步修改 Keil 工程，则调整虚拟分组和输出路径。
11. 如果用户选择添加工具源码，则优先运行 `scripts/copy_utilities.ps1`，将 `assets/utilities/` 下的文件复制到目标工程的 `Common/utilities/`，不要只依赖手工复制。
12. 将 `assets/README-template.md` 复制到目标工程根目录，并命名为 `readme.md`，再根据实际工程做轻量调整。
13. 如果用户需要 Keil 配置教学，则使用 `assets/keil-config-guide/` 中的图片资源。
14. 询问用户 Keil 编译工具是否已经加入系统环境变量或 `PATH`。
15. 如果用户回答“是”，则不要直接假设可用，必须先在 PowerShell 中用 `Get-Command UV4.exe` 进行检查；必要时继续检查 `UV5.exe`、`uvision.exe` 等其他可能的 Keil 可执行文件名，以确认解析出来的真实路径。
16. 如果用户回答“否”，则必须要求用户提供编译工具目录或可执行文件的绝对路径，并给出示例路径，例如 `D:\ruanjian\keil5-MDK-ARM\UV4\`，但不要假设用户的工具一定就是 `UV4`。
17. 在 Keil 目录结构和工程配置修改完成后，只要环境支持命令行执行，就应基于已确认的编译工具路径做一次编译验证。

## 必须确认的结构化选项

下列选项属于必须确认项。在运行环境支持结构化选择框时，必须优先以可点击选项的形式向用户展示，不要默认跳过，也不要直接自行假设。

1. 确认项目是否为单工程结构。
   - 继续
   - 退出
2. 确认用户是否知晓：此 skill 会同时修改本地目录结构以及 Keil 工程结构或相关配置。
   - 继续
   - 退出
3. 如果源码中原本没有 `Common/utilities` 或等价的工具类源码目录，则确认是否要从 `assets/utilities/` 复制工具源码。
   - 是
   - 否
4. 确认是否附带 Keil 输出路径和 `fromelf` 配置教学资源。
   - 是
   - 否
5. 如果上下文中尚不明确，则确认当前属于“新建工程结构”还是“修改已有工程”。
   - 新建工程
   - 修改已有工程
6. 确认输出方式是“保留原工程备份后修改”还是“另起一个新目录输出”。
   - 保留备份后修改
   - 输出到新目录
7. 询问用户 Keil 编译工具是否已经加入系统环境变量或 `PATH`。
   - 是
   - 否
   - 如果回答“是”，先在 PowerShell 中用 `Get-Command UV4.exe` 检查；必要时继续检查其他可能的 Keil 可执行文件名，以确认真实路径。
   - 如果回答“否”，必须要求用户提供编译工具目录或可执行文件的绝对路径，并给出示例路径，例如 `D:\ruanjian\keil5-MDK-ARM\UV4\`，但不要假设用户的工具一定就是 `UV4`。

## 核心约束

- 允许修改目录结构和工程配置文件。
- 不允许改动业务源码逻辑。
- 根目录名保持不变。
- 仅支持单工程 Keil 结构。
- Keil 分组是虚拟分组，不要求与本地目录完全同名。
- 如果运行环境支持结构化选择框，则优先采用点击式选项交互。
- 只有在不支持结构化交互工具时，才退化为文本选择题。
- “必须确认的结构化选项”属于强制确认项，不是可选建议项。
- `startup_xxx.s`、`.sct` 这类工具链相关文件应放在 `Project/Keil-MDK/`。
- 不要主动改写 `Drivers/` 下的官方底层驱动内容，除非用户单独明确提出。
- 不要因为用户说编译工具已经加入 `PATH` 就直接假设可调用，必须先用 PowerShell `Get-Command` 验证。
- 如果编译工具没有加入 `PATH`，则在尝试编译验证前必须向用户索取绝对路径。
- 只要已经拿到可用的编译工具路径，且当前环境支持命令行执行，就应在重组完成后追加一次编译验证。
- 在检查编译工具时，除了 `UV4.exe`，也要根据实际情况检查 `UV5.exe`、`uvision.exe` 等其他可能的 Keil 可执行文件名，不要固定假设只有一个文件名。

## 资源说明

- `assets/templates/`
  - 存放本地目录树模板和 Keil Group 模板。
- `assets/README-template.md`
  - 用于生成目标工程根目录的 `readme.md`。
- `assets/gitignore/keil-template.gitignore`
  - 用于生成匹配该目录结构的 `.gitignore`。
- `scripts/copy_utilities.ps1`
  - 用于将 `assets/utilities/` 下的工具源码确定性复制到目标工程的 `Common/utilities/`。
- `assets/utilities/`
  - 存放可选复制的通用工具源码。
- `assets/keil-config-guide/`
  - 存放 Keil 输出路径和 `fromelf` 配置相关截图。
