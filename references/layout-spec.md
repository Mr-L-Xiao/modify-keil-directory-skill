# Layout Specification

This reference defines the standardized local directory layout and Keil group layout for single-project MCU + Keil projects, with or without RTOS.

## Base Local Layout

```text
App/
  function/
  User_Config/
  main.c
BSP/
  board/
  devices/
Common/
  User_System/
  utilities/
Docs/
Drivers/
Output/
  Listings/
  Objects/
  Bin/
Project/
  Keil-MDK/
```

- `App/tasks/` exists only in RTOS projects.
- `Middlewares/FreeRTOS/` exists only in RTOS projects.

## Base Keil Group Layout

```text
项目名
├─System
├─App/User
├─App/function
├─Common
├─BSP/board
├─BSP/devices
├─Drivers/xx32xx_Driver
└─...
```

- This is the common base for both RTOS and bare-metal projects.
- RTOS projects add `App/tasks` and `Middlewares/FreeRTOS`.
- Keil groups are virtual groups and do not need to match local directory names exactly.

## Directory Responsibilities

- `App`
  - Application layer: entry point, tasks, business flow, state machines.
  - `function/`: business function code, and optionally peripheral-related code that already contains business logic.
  - `tasks/`: RTOS task code only.
  - `User_Config/`: user-defined configuration files, including runtime and compile-time global config `.c` and `.h` files.
  - `main.c`: application entry point.

- `BSP`
  - Board support package.
  - `board/`: board-level peripheral support code such as `bsp_gpio.c`, `bsp_adc.c`.
  - `devices/`: user-written peripheral hardware control code without business logic, such as `led`, `motor`, `encoder`, or other sensors.

- `Common`
  - Common system glue and shared code.
  - `User_System/`: platform glue files such as `xx_it.c/h`, `interrupts_xx32xx.c/h`, `systick.c/h`.
  - `utilities/`: utility code such as `crc16.c/h`, `ring_buffer.c/h`.

- `Docs`
  - Project documentation.

- `Drivers`
  - Vendor/CMSIS/low-level driver and device support files.
  - `CMSIS/`: CMSIS-provided content.
  - `xx32xx_Driver/`: vendor low-level peripheral driver library.
  - `xx32xx_System/`: chip or vendor support system files such as `hc32l19x.h`, `stm32f1xx.h`, `system_xxx.c`.

- `Middlewares`
  - Third-party middleware source code.
  - `FreeRTOS/`: RTOS source tree and `FreeRTOSConfig.h`.

- `Output`
  - Build outputs.
  - `Listings/`: text reports such as `.map` and `.lst`.
  - `Objects/`: `.o`, `.axf`, `.hex`, and related build outputs.
  - `Bin/`: `.bin` files generated by `fromelf`.

- `Project`
  - Project files and toolchain-specific startup/linker files.
  - `Keil-MDK/`: `uvprojx`, startup assembly, `.sct`, and related Keil files.

## File Placement Rules

- `startup_xxx.s` and `.sct`
  - Place under `Project/Keil-MDK/`.
  - These are toolchain-related files, not shared source files.

- `hc32l19x.h`, `stm32f1xx.h`, `core_cm*.h`, `system_xxx.c`
  - Place under `Drivers/xx32xx_System/` or keep within `Drivers/CMSIS/` if already packaged there.

- `xx_it.c`, `xx_it.h`
  - Place under `Common/User_System/`.
  - These are project-level interrupt wiring files and often contain user modifications.

- `systick.c`, `systick.h`
  - Place under `Common/User_System/`.
  - These are user-defined system timing or delay wrappers.

## RTOS Keil Group Layout

```text
项目名
├─System
│  ├─startup_xx32xx.s
│  ├─stm32f1xx.h
│  ├─system_xxx.c
│  ├─xx32xx_it.c
│  └─...
├─App/User
│  ├─configure.c
│  └─main.c
├─App/tasks
│  ├─key_task.c
│  ├─uart_task.c
│  └─...
├─App/function
│  ├─key_func.c
│  ├─uart_func.c
│  └─...
├─Common
│  ├─crc16.c
│  ├─ring_buffer.c
│  └─...
├─BSP/board
│  ├─bsp_gpio.c
│  ├─bsp_adc.c
│  └─...
├─BSP/devices
│  ├─motor.c
│  ├─encoder.c
│  └─...
├─Drivers/xx32xx_Driver
│  ├─stm32f1xx_gpio.c
│  ├─stm32f1xx_adc.c
│  └─...
├─Middlewares/FreeRTOS
│  ├─list.c
│  ├─heap_4.c
│  └─...
```

Notes:

- `System` contains startup files plus files from `Drivers/xx32xx_System` and `Common/User_System`.
- `Common` group contains files from `Common/utilities`.
- Add `App/tasks` only when RTOS is used and task count is large enough to justify a separate group.

## Bare-Metal Local Layout

Use the base layout and remove RTOS-only directories:

```text
App/
  function/
  User_Config/
  main.c
BSP/
  board/
  devices/
Common/
  User_System/
  utilities/
Docs/
Drivers/
Output/
  Listings/
  Objects/
  Bin/
Project/
  Keil-MDK/
```

- Do not create `App/tasks/`.
- Do not create `Middlewares/FreeRTOS/`.
- `main.c` enters the main loop or event loop after initialization.
- Bare-metal business logic still belongs in `App/function/` and `Common/User_System/` where appropriate.

## Bare-Metal Keil Group Layout

```text
项目名
├─System
├─App/User
├─App/function
├─Common
├─BSP/board
├─BSP/devices
├─Drivers/xx32xx_Driver
└─...
```

- Do not create `App/tasks`.
- Do not create `Middlewares/FreeRTOS`.

## Keil Output Path Guidance

- `Objects`
  - Store `.o/.axf/.hex` and similar outputs.
- `Listings`
  - Store `.map/.lst` and similar analysis outputs.
- `Bin`
  - Store `.bin` generated by `fromelf`.

Keil does not directly emit `.bin`, so use an After Build command such as:

```text
fromelf --bin -o "..\Output\ProjectName.bin" "..\Output\Objects\ProjectName.axf"
```

This layout helps with Git management:

- Ignore most intermediate outputs under `Output/Objects/` and `Output/Listings/`.
- Keep or ship `.bin` under `Output/Bin/` only if the workflow benefits from that.

## Build Tool Detection and Compile Verification

- Before applying changes, ask whether the Keil build tool has been added to system `PATH`.
- If the user answers `Yes`, first verify the resolved path in PowerShell with `Get-Command UV4.exe`, and check other likely executable names such as `UV5.exe` or `uvision.exe` when needed.
- If the user answers `No`, require the user to provide the absolute path to the build tool directory or executable. Show an example such as `D:\ruanjian\keil5-MDK-ARM\UV4\`, but do not assume the tool is necessarily `UV4`.
- After restructuring, use the resolved build tool path to run a compile verification when command-line execution is available in the environment.

## Interaction Checklist

Treat the following as required confirmations before applying changes. If the runtime supports structured clickable choices, use them instead of free-form text:

1. Whether the target is a single-project Keil workspace.
   - Continue
   - Exit
2. Whether the user wants to continue after being reminded that both the local directory structure and the Keil project structure or configuration may be modified.
   - Continue
   - Exit
3. Whether the project uses RTOS.
4. Whether this is a new project or a restructuring of an existing one.
   - New project
   - Modify existing project
5. Whether to keep a backup or generate a new output directory.
   - Keep backup / modify with backup
   - Generate new directory
6. Whether to copy utility files from `assets/utilities/` when the project does not already contain an equivalent utilities area.
   - Yes
   - No
7. Whether to include Keil output configuration guidance assets.
   - Yes
   - No
8. Whether the Keil build tool has been added to system `PATH`.
   - Yes
   - No
   - If yes, first verify it with `Get-Command UV4.exe` and other likely Keil executable names as needed.
   - If no, require the user to provide an absolute path such as `D:\ruanjian\keil5-MDK-ARM\UV4\` before compile verification.

## Constraints

- Modify project layout and project configuration files, but do not change business source logic.
- Keep the root directory name unchanged.
- Support only single-project structures.
- Top-level local folder names follow the standardized naming convention from this layout.
- Prefer copying to a new output directory or keeping a backup when restructuring an existing project.
- Do not modify official vendor files under `Drivers/` unless explicitly requested.
- If a project has no true `BSP/board` content and only user device-control code, it is acceptable to omit `BSP/board` and keep only `BSP/devices`.
