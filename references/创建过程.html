                    其中各个文件夹的作用及存放内容如下：

                    • App：应用层：main、任务、业务流程、状态机

                      • function：存放业务功能代码（主要供 task 调用），也可以放包含业务逻辑的外设开发代码；

                      • tasks：存放应用任务代码（要用 RTOS 才会有此文件夹）

                      • User_Config：存放用户自定义配置文件（包括运行时和编译时的全局配置.c和.h文件）

                      • main.c：应用入口函数

                    • BSP：板级支持包

                      • board：存放板级外设支持包（比如 bsp_adc.c、bsp_adc.h、bsp_gpio.c……）

                      • devices：存放用户自己写的外设硬件控制代码（不含业务逻辑，例如led/电机/其他传感器等）

                        • led：存放led相关的.c和.h文件

                        • sensors：……

                        • ……

                    • Common：存放代码框架系统胶水与资源文件

                      • User_System：存放平台级胶水文件（xx32xx_it.c/h 或 interrupts_xx32xx.c/h）以及用户自己写的通用源文件（比如 systick.c/h 之类）

                      • utilities：存放通用的工具代码（crc16.c、crc16.h……）

                    • Docs：存放工程相关的文档

                    • Drivers：存放芯片厂商/CMSIS/底层驱动与设备支持文件

                      • CMSIS：……

                      • xx32xx_Driver：存放主控芯片的底层驱动库（一般由官方提供）

                      • xx32xx_System：存放芯片/厂商支持系统文件（hc32l19x.h、stm32f1xx.h、system_xxx.c……）

                    • Middlewares：存放第三方中间件源码

                      • FreeRTOS：若有 FreeRTOS 之类的实时操作系统源代码，就存放于此，FreeRTOSConfig.h 也放在这里

                      • ……

                    • Output：存放编译产物（下面会教如何配置输出路径）

                      • Listings：存放编译器生成的文本类分析报告（包括 .map 内存映射文件和 .lst 汇编列表文件等）

                      • Objects：存放编译链接全流程的产物（包括 .o/.axf/.hex 等）

                      • Bin：存放 .bin 文件（在 Keil 的 User 选项卡中使用 fromelf 指令生成）

                    • Project：存放工程文件

                      • Keil-MDK：存放 keil 工程文件

                    额外说明：

                    Drivers/xx32xx_Driver 下面的文件放的是芯片/厂商级通用底层驱动与设备支持文件，主要是对寄存器操作做的一层封装，没有规定哪个 gpio 是用来干啥的

                    • 告诉用户“主控芯片一般能做什么”

                    • 比如“读取某个寄存器来获得 GPIO 输入电平”

                    BSP/board 下面的文件放的是板级支持包，主要是对板级外设的配置，并且开始涉及一些简单的逻辑判断，但仍不涉及业务逻辑

                    • 告诉用户“在这个板子上，这个主控芯片的哪些部分怎么用”

                    • 比如“初始化某个 GPIO”

                    mcu 的启动文件放在工程文件夹下面：startup_xxx.s，放 Project/Keil-MDK/。.sct 这类链接脚本也放 Project/Keil-MDK/，不过一般不会手动写连接脚本，都是编译器内置的。

                    • startup_xxx.s 和链接脚本是“工具链相关文件”，不是通用源代码，所以不放 Drivers、也不放 Common，直接跟着 Keil 工程走最稳。

                    hc32l19x.h、stm32f1xx.h、core_cm*.h、system_xxx.c：放 Drivers/xx32xx_System/

                    • hc32l19x.h、system_hc32l19x.c 这类是“芯片/厂商支持文件”，本质上属于底层设备支持包，应该放 Drivers

                    • 如果源码中已经有 CMSIS 文件夹了，那这个文件夹的内容就不要动，继续放在 CMSIS 里；如果有其他在外面的文件，就放进 Drivers/xx32xx_System/ 中。

                    xx_it.c、xx_it.h：放 Common/User_System/

                    • xx32_it.c/h 或 interrupts_xx32xx.c/h 是“工程级中断接线文件”，通常会被用户修改，里面还会调用项目里的函数，所以更适合放 Common/User_System/，不建议塞进 Drivers

                    • 启动文件中有中断向量表，其中规定了所有外设的中断名，比如 UART0_IRQHandler。然后会在类似 stm32f1xx_it.c、interrupts_hc32l19x.c 的中断接线文件中定义对应的中断入口函数（中断服务函数）。实际使用分两种情况：

                      • 简单的标准库工程可以直接在这些接线文件中的中断函数下直接写业务逻辑代码

                      • 复杂的标准库工程往往是在这些函数下又调用一层函数，这一层函数是由 __WEAK 修饰的，表示可被修改的弱定义（但函数名、入参/出参不能改，只能改函数内容），然后用户可以在自己的业务层代码文件下重定义这些函数，然后写入业务逻辑代码

                      • 但像 STM32 的 HAL 库工程：多了一些封装、多了一些回调，但本质是一样的

                    systick.c、systick.h 之类的用户自己写的系统级文件：默认也放 Common/User_System/

                    • systick.c/h 如果是用户自己写的系统节拍、延时、超时基准封装，它和 it.c/h 一样，也属于“工程公共系统胶水”，放 Common/User_System/ 最稳

                    User_Config 文件夹下放用户自己定义的配置文件，可以包含运行时/编译时的全局配置 .c 和 .h 文件。

                    需要和用户交互的点

                    更倾向于给用户出选择题，最好是有一个选项框，可以让用户点击选项，然后再点确认后进行下一步

                    • 先提醒用户此 skill 暂时只能用于单工程结构，如果项目里有 boot、master 这类多工程结构，则无法改动。所以在使用前要让用户选择是否为单工程结构，是则继续，否则退出。

                    • 并且由于本地目录修改了，所以源代码中的 .uvprojx 这些类似的工程文件可能也需要修改，主要是修改其中的配置项；修改前提醒用户：“此 skill 会同时修改本地目录和 keil 工程结构，请仔细查看源代码结构，并决定是否继续修改”，两个选项（继续 / 退出）。

                    • 如果源码中没有 Utilities 文件夹下的文件，要在对话窗口让用户做选择是否要添加进去，两个选项（是 / 否），这些都会放在 assets 里。

                    • 调用此 skill 之前问一下用户，是否用了 RTOS，两个选项（是/否）。

                    • skill 相关文件中也会放对应的 keil 配置教学，要在对话窗口让用户做选择是否要添加进去，两个选项（是 / 否）。

                    • 如果在创建前不确定是“新建工程”还是“修改已有工程”，先让用户选一个，再继续。

                    • 如果要保留原工程备份，或者另起一个新目录，也要先问用户选哪种输出方式。

                    注意项

                    • 使用此 skill 将本地的目录结构修改之后，要在根目录生成一个 readme.md，要讲解一下目录结构和文件功能。

                    • 此 skill 允许修改工程配置文件和说明文件，但不允许改业务源码内容！！！！

                    • 在利用这个 skill 修改其他 keil 本地文件夹结构时，根目录名保留，只更改根目录下的子文件夹。

                    • 目前只针对单工程，即不包含 boot、master 等多工程。

                    • 根目录下的一层文件夹命名都遵循：首字母大写，其他小写。

                    • 此 skill 在调用的时候，不要直接替代源代码，要复制并重新生成一个新目录结构（主要是防止给用户改错了，备份一下）。

                    • Drivers 下面的文件是不可以被用户修改的，是由官方提供的底层库/底层系统配置文件。

                    • 有的源代码只有底层寄存器驱动库（例如 STM32F1xx_HAL_Driver），没有 BSP/board 板级支持包。如果遇到这样的情况，可以删除 BSP/board 文件夹，只留下 BSP/devices 文件夹存放用户自己写的外设控制代码。

                    • Keil 分组是虚拟分组，不要求必须与本地目录同名，尽量按照之前的大纲来。
